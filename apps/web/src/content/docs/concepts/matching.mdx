This is the single most important design decision in Tactile, and the thing the popular incumbents
get wrong by leaning entirely on the deprecated `keyCode`.

The engine keeps two ideas separate: the **character** a key produces (`KeyboardEvent.key`) and its
**physical position** (`KeyboardEvent.code`). Each binding token is matched by one or the other,
chosen by the match mode.

| Mode | Letters & digits | Named keys (Enter, ‚Üë, F5) | Symbols (`?`, `/`) |
| --- | --- | --- | --- |
| `hybrid` *(default)* | physical (`code`) | logical (`key`) | logical (`key`) |
| `physical` | physical | physical | falls back to `key` |
| `logical` | logical | logical | logical |

## Why hybrid defaults the way it does

- **Letters ‚Üí physical.** `‚åòK` should be the K *position* ‚Äî that's what app authors almost always
  mean, and it stays put when the layout changes the character. On a German layout where the physical
  `KeyZ` produces "y", `‚åòZ` still fires on the Z key. `keyCode`-based libraries get this wrong.
- **Symbols ‚Üí logical.** You mean the glyph `?`, whose physical key differs by layout. Matching the
  character is the only thing that behaves internationally.
- **Named keys ‚Üí logical.** `Enter`, `ArrowUp`, `F5` have stable, layout-independent `key` values.

Read [International keyboards](/concepts/international-keyboards/) for the Dvorak/keycap convention and help UI implications.

## Configure match mode

Globally:

```ts
const kb = createKeybindingEngine({ defaultMatch: 'hybrid' });
```

Per rule:

```ts
kb.add({ id: 'vim.down', keys: 'j', match: 'physical', handler: down });
kb.add({ id: 'search', keys: 'mod+k', match: 'logical', handler: search });
```

## Modifier matching

Ctrl, Alt, Meta are matched **exactly**. Shift is matched exactly **except** for symbol main keys:
producing `?` already requires Shift on many layouts, so enforcing Shift in the binding would make
`'?'` impossible to register.

```ts
// These are equivalent on US QWERTY for the ? key:
kb.add({ id: 'help', keys: '?', handler: help });
// User presses Shift+/ ‚Üí event.key is '?', matches
```

## How matching works at runtime

For each `keydown` (or `keyup` if `eventType: 'keyup'`):

1. **Ignore check** ‚Äî skip if focused in an input (configurable).
2. **Context check** ‚Äî evaluate each rule's `when` against `engine.context`.
3. **Chord match** ‚Äî compare modifiers + main key using the rule's match mode.
4. **Sequence check** ‚Äî for `g i`, match the last N keystrokes within `sequenceTimeout`.
5. **Collision resolve** ‚Äî highest `priority`, then last registered wins.
6. **Handler** ‚Äî call with `(event, { id, keys, sequence })`.

## Test cases (what we verify)

```ts
// German QWERTZ: physical Z produces 'y'
event = { key: 'y', code: 'KeyZ', ctrlKey: true }
match('ctrl+z', event, 'hybrid')  // ‚úÖ true  ‚Äî Z keycap
match('ctrl+y', event, 'hybrid')  // ‚ùå false ‚Äî not the Y keycap

// Dvorak: physical K produces 't'
event = { key: 't', code: 'KeyK', metaKey: true }
match('mod+k', event, 'hybrid')   // ‚úÖ true  ‚Äî K keycap

// Symbol: character matters
event = { key: '?', code: 'Slash', shiftKey: true }
match('?', event, 'hybrid')       // ‚úÖ true
```

## Lower-level API

For custom tooling, use `matchChord` directly:

```ts
import { parseBinding, matchChord } from '@tactile-js/core';

const chord = parseBinding('mod+z', 'other')[0];
matchChord(chord, keyboardEvent, 'hybrid');
```
